## Recommendation

Do not add another package surface yet. The strongest near-term candidate is a comparison-driven CI regression recipe, but it should remain a monitored follow-up until one downstream repo proves repeated setup friction after adopting `batch-review-compare --json` and `review-gate --batch-comparison`.

The explicit extraction question is whether a future helper can stay at the generic package boundary: placeholder manifests, caller-owned budgets, sanitized artifacts, and no private workflow labels. The capture-review adapter contract records this as a manifest-shaped boundary rather than a new wrapper. If the next adoption attempt needs repo-specific naming, private capture semantics, or domain policy, keep that wrapper in the embedding repo instead of promoting it into `ai-quality-loops`.

For claim-risk checks, the generic-vs-domain-specific extraction question is whether the missing reusable piece is a neutral contract dimension such as evidence traceability or uncertainty handling, rather than venture policy, legal posture, reputation thresholds, buyer context, or company-specific proof standards. If the latter is needed, keep the checker in the embedding repo as a caller-owned injection over the published high-stakes rubric.

For context-pack quality checks, the generic-vs-domain-specific extraction question is whether repeated adopters need the same neutral rubric wording for source traceability, staleness risk, contradiction handling, and authority-boundary clarity. Source discovery, source authority, freshness policy, pack assembly, truth reconciliation, and action routing stay outside AIQL.

For packet verification-command fit checks, the generic-vs-domain-specific extraction question is whether repeated adopters need the same neutral rubric wording for repo-native validation ownership, stale command assumptions, and explicit blocker language. Repository inspection, package-manager choice, hook policy, launcher behavior, tracker state, and enforcement stay outside AIQL.

For public-facing creative support reviews, the generic-vs-domain-specific extraction question is whether repeated adopters need the same neutral rubric wording for clarity, support, audience fit, accessibility, and authority boundaries. Brand context, final taste, market positioning, publication approval, escalation thresholds, and action routing stay outside AIQL.

For sanitized social evidence reviews, the generic-vs-domain-specific extraction question is whether repeated adopters need the same neutral rubric wording for proportional claims, caveat visibility, source traceability, and authority boundaries. Real account data, audience strategy, proof thresholds, publication approval, and business routing stay outside AIQL.

For launch-outcome evidence summarization, the generic-vs-domain-specific extraction question is whether repeated adopters need the same neutral compaction format over AIQL JSON artifacts, or whether the missing value is launch policy, tracker semantics, source selection, readiness labels, or action routing. If the latter is needed, keep the summarizer in the embedding repo and use AIQL only for sanitized review outputs, comparisons, gates, and documentation-only fixture shapes.
